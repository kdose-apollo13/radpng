CORE SHAPE
- One atom, one file. One job per export when that job is independently meaningful.
- Name exports at the top (`__all__` or a one-line manifest comment).
- Atoms before monoliths. Composition before inheritance.
- Orchestrate at the edges; keep the core dumb.
- Explicit beats clever. Plain loops beat magic.
- Bytes beat objects; dicts beat classes — unless the GUI demands otherwise.
- Read each file top to bottom. If you cannot, split it.
  (If splitting hurts semantic coherence or equal-signature swap, name the cost and stop.)

CONTRACTS
- `: ` for inputs. `> ` for outputs. `! ` for what must not happen.
- Let there be a blank line between the sections :, > and !
- Say the contract in the docstring; skip the sermon.
- Raise clear, specific exceptions for caller errors (ValueError for bad inputs).
  Put a short description of the sin in the message.
- In tests: `self.rais(Exc)` uniformly; `with self.rais(Exc) as ctx` + `self.isin` for messages.
- NotImplemented is honest refusal, not shame. Admitted tradeoffs are kept tradeoffs.

STDLIB & DEMOS
- Pure stdlib only. No deps, no drama. -> hacker ethos, old school,
- Every module is a demo: `if __name__ == '__main__'` proves the atom.
- Thin library, thick tests. Test-only helpers stay in tests.
- Pack what callers must pack; do not smuggle convenience into the surface.

TESTS
- One test file per major atom or semantic facility.
- Given, When, Then — every scenario speaks plain English.
- Short aliases in tests; long names in libraries.
- Symmetry proves correctness: encode/decode, filter/unfilter. Roundtrip is the oracle.

POWER LAW OF MODULE SIZES
- Most files are small focused atoms (leaves).
- A modest number of unified semantic facilities are larger — they group what is
  understood, varied, or demoed together (row geometry, filter transform, PNG pipeline,
  viewer GUI cluster, etc.).
- Bias small when a clean single job exists.
- Design for the power law that serves: semantic coherence, low import explosion,
  easy demo at a useful abstraction, swappability during optimization.
- When a rule feels ridiculous, try the split (or consolidation) anyway. Observe density
  effects on correctness, comprehension, and variation. Name over/under-fragmentation. Tune.

GUI CLUSTER (viewer lesson — tkinter is the justified class exception)
- Split ruthlessly: widgets (canvas, cmd bar), focus state machine, command parse,
  window geometry, pure image rows. Orchestrator wires; atoms do not know verbs.
- Tk focus traversal is the enemy. Set `takefocus=0` on chrome (frames, scrollbars,
  labels). Only intentional targets take focus. Drive Tab/Esc/: with an explicit state
  machine atom — never rely on default widget tab order.
- Three modes when interaction matters: content (img), command (cmd), general (window).
  Tab cycles content <-> command only. Esc -> general. `:` from content or general -> cmd.
- Commands are plain text after `:` (no leading colon in the string). Parse in a pure atom;
  apply window verbs in another; keep PhotoImage glue in the orchestrator.
- GUI atoms demo alone; pure atoms get unit tests. Beautiful tests, boring code.

DISCIPLINE
- Radical is not reckless. Decompose, march, accept the cost.
- Some generality deferred is complexity slain.
- Informative failures, quiet successes. When all pass: Radical!
- Duplication in 2nd-order tests may be the honest price of isolation.
